# 前言：经纬之间

> **「经之以天，纬之以地。」**——《左传·昭公二十八年》
>
> 两千年前，织工以经线为骨、纬线为肉，一梭一梭织就锦缎。经，是结构——纵贯始终、张紧不移；纬，是功能——穿梭其间、变化万千。经定其形，纬成其华；经纬交织，方成一匹布。
>
> 今天，编排 AI Agent 亦复如是：`meta` 与 `phase` 是**经**——确定性的结构骨架，预先张紧、不可动摇；`agent()`、`parallel()`、`pipeline()` 是**纬**——在骨架中穿梭执行的智能单元。经线决定流水线的形状，纬线填入真正的工作。
>
> 本书因此得名 **织经**。

## 当所有人都在「指挥」Agent

过去两年，我们学会了**使用** AI Agent：写好提示词、调好工具、开几个子任务并行跑一跑。社区里涌现出大量优秀的工作流系统——`oh-my-claudecode`、`superpowers`、`oh-my-openagent`、`ccg-workflow`——它们各显神通，把单个 Agent 调教成一支训练有素的团队。

但如果你拆开它们的引擎盖，会发现一个共同的、略显尴尬的事实：**这些系统都在用提示词「祈祷」式地编排。**

它们把编排逻辑写在 Markdown 里，靠在提示词中反复强调「⛔ 你必须先做 A 再做 B」「绝不允许跳过验证」来约束一个**概率性**的语言模型；它们用生命周期钩子（hooks）在每一轮对话里注入「面包屑」，提醒 Agent「你还没干完，回去接着干」；它们把进度落盘成 JSON 状态文件，因为一旦上下文被压缩，Agent 就会忘记自己在做什么。

这些技巧很聪明，也很有效——但它们的本质，是在用**自然语言**和**运行时补丁**，去模拟一个本该由**代码**保证的东西：**确定性的控制流**。

> 它们之所以这样做，是因为在很长一段时间里，Claude Code 并没有提供「用代码编排 Agent」的原生能力。
>
> 现在，它提供了。

## CLAUDE_CODE_WORKFLOWS：被悄悄加入的确定性引擎

Claude Code 在 `CLAUDE_CODE_WORKFLOWS` 这个功能标志（feature flag）背后，藏着一个叫 **Workflow** 的内置工具。它做的事情，一句话概括就是：

> **用一段纯 JavaScript 脚本，确定性地编排任意多个 subagent——支持流水线、并发、阶段、预算、结构化输出与 JSON Schema 约束，并且可复用、可测试、可分享。**

这不是 MCP（那是连接外部工具的协议），不是 Skills（那是注入提示词的知识包），不是 Subagents（那是一次性的子任务），也不是 Agent Teams（那是有状态的协作团队）。它是一个**全新的、正交的扩展维度**：把「先做什么、再做什么、哪些并行、哪些串行、出了结果怎么验证」这套**编排逻辑**，从飘忽的提示词里，搬进了**确定性的代码**里。

当你写下：

```javascript
const results = await pipeline(
  dimensions,
  d => agent(d.reviewPrompt, { schema: FINDINGS }),
  review => parallel(review.findings.map(f => () =>
    agent(`对抗性验证这条发现：${f.title}`, { schema: VERDICT })
  ))
)
```

`pipeline`、`parallel`、`agent` 都是**真实的函数**，由 JavaScript 运行时**确定性地**执行。哪个阶段先跑、几个 agent 并发、循环到什么条件退出——全部由代码说了算，不再依赖模型「自觉」。而 `schema` 会强制每个 subagent 的产物**严格匹配** JSON Schema，模型若返回不合规的结构，运行时会让它**重试**，直到合规为止。

这意味着：**社区那些靠提示词苦苦维系的编排纪律，现在可以用代码一次性焊死。**

## 这本书是什么，不是什么

**这是一本 Cookbook（食谱书），不是 API 文档。**

市面上不缺「Workflow 工具有哪些参数」的罗列。本书要回答的是更难、也更有用的问题：

- **何时**该用 Workflow，何时该用 Subagents / Skills / Agent Teams？（第一部 · 定位矩阵）
- `parallel` 和 `pipeline` 看起来都能并发，**到底差在哪**，选错会浪费多少墙钟时间？（第二部 · 基础篇）
- 一个真正能用的「分片代码审查」「PR 多维评审」「Bug 猎手」流水线，**长什么样**？（第三部 · 实战食谱）
- 怎样设计「对抗验证」「评委面板」「循环到干」，让结果**可信**而不只是「看起来对」？（第四部 · 进阶模式）
- 那四个优秀的社区系统，**哪些精华**可以用 Workflow 重写成可复用的资产？（第五部 · 生态与借鉴）
- 怎样从零搭建**属于你自己**的、可复用、可分享的 Workflow 库？（第五部 · 构建你的库）

## 三个不容妥协的承诺

本书与许多「AI 写的教程」最大的不同，在于三条铁律：

**一、真实运行，绝不伪造。** 书中每一段标注「真实运行」的输出，都来自在真实 Claude Code 会话中实际执行 Workflow 所得的原始结果——包括真实的 `taskId`、`runId`、token 用量、耗时与返回值。这些原始记录保存在仓库的 [`assets/transcripts/`](https://github.com/AGI-is-going-to-arrive/workflow-cookbook/tree/main/assets/transcripts) 目录，可逐条核对。凡是未实际运行、仅作示意的脚本，都会**明确标注**。

**二、信源对照，绝不臆测。** 所有关于 Workflow API 的描述，都对照 Claude Code 官方分发包中的类型定义文件 `sdk-tools.d.ts`（`WorkflowInput` / `WorkflowOutput` 接口）与运行时的工具定义逐字核对。涉及环境变量、版本号、功能标志的论断，均经本机实测确认。

**三、口径一致，中英对照。** 本书提供完整的中英双语版本，两种语言一一对应、术语统一。你在任意章节点击右上角的语言切换，都会跳到同一章的另一语言版本。

## 实测环境声明

> 本书所有实测均在以下环境完成，请在阅读配方时以此为基准：
>
> | 项 | 值 |
> |---|---|
> | Claude Code 版本 | **v2.1.150**（原生二进制） |
> | 功能标志 | `CLAUDE_CODE_WORKFLOWS=1`（已确认存在于会话环境） |
> | 主模型 | Opus 4.7（1M 上下文） |
> | subagent 模型 | `claude-opus-4-7`（由 `CLAUDE_CODE_SUBAGENT_MODEL` 指定） |
> | 关联标志 | `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` |
> | 实测时间 | 2026 年 5 月 |

> Workflow 仍是一个**实验性、需显式开启**的功能。不同版本的具体行为（并发上限、预算语义、续传细节）可能演进。书中会标注关键行为的来源，方便你在自己的版本上复核。

> **赶时间？** 第 01 章建立认知 → 第 04 章跑通第一个 → 第 08 章搞懂 `parallel` vs `pipeline` → 挑一个第三部的配方照着改。
